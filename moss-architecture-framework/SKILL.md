# 软件架构 · 思维操作系统

> 「复杂性有利息。你不付它，它就拖垮你。」——Ward Cunningham / Martin Fowler

## 框架概览

### 这个框架解决什么问题

当你面对以下问题时，这个Skill可以作为思考顾问：

| 问题类型 | 典型问法 |
|---------|---------|
| **架构拆解** | 「这个大系统怎么拆」「服务边界怎么划」「这个模块边界合理吗」 |
| **需求分解** | 「这个模糊需求怎么落地」「怎么拆成可执行的任务」「先做哪个」 |
| **团队协同** | 「团队怎么分工」「依赖怎么管」「谁做决策」 |
| **架构审视** | 「这个架构有什么问题」「会不会过度设计了」「还差什么」 |

### 核心工作流

| Phase | 输入 | 工具 | 输出 |
|-------|------|------|------|
| **Phase 1: 需求理解** | 模糊的产品需求、用户反馈 | Impact Mapping / User Story Mapping | 清晰的用户故事、优先级排序 |
| **Phase 2: 架构拆解** | 用户故事、业务域划分 | Conway's Law + Bounded Context + 模块化原则 | 服务边界、模块划分、架构视图 |
| **Phase 3: 任务分解** | 服务边界、模块定义 | INVEST + MoSCoW + Story Splitting | 可执行的Task列表、 Sprint规划 |
| **Phase 4: 团队协同** | Task列表、依赖关系 | Team Topologies + WIP Limits + ADR | 团队分工、协作接口、决策记录 |
| **Phase 5: 持续演进** | 运行中的系统、监控数据 | 技术债务篱笆 + ADR + 演化式设计 | 重构计划、架构演进记录 |

**流程图（含决策节点）**：
```
[Phase 1] → [Phase 2] → [Phase 3] → [Phase 4] → [Phase 5]
   ↑            ↓↑                         ↓↑
   │      ⚠️团队重组确认点               ⚠️架构决策确认点
   │            ↓                          ↓
   │      [H6边界检验] ←────────────────→ [H4债务围栏]
   │
   └──────────────── 反馈循环 ←────────────────┘

⚠️ = 决策确认节点（必须停下来确认）
```

### 条件分支

| 条件 | 路径 |
|------|------|
| **需求已经很清楚** | Phase 1 → 直接跳Phase 2 |
| **团队很小（<10人）** | Phase 1+2+3可以合并快速做 |
| **架构已经在运行** | 从Phase 4进入，Phase 5随时可返回Phase 2 |
| **微服务拆分** | 必须先完成Phase 2边界定义，再进Phase 3 |
| **遗留系统重构** | Phase 5先行（技术债务篱笆），再返回Phase 2 |

### 快速入口

如果你已经知道问题类型，直接跳到对应框架：

| 已知问题 | 直接跳转 |
|---------|---------|
| 团队要重组 | → 框架2 + 决策确认节点 |
| 边界模糊 | → H6分解边界检验 |
| 要拆微服务 | → ⚠️决策确认节点 → H7 |
| 方案犹豫不决 | → ⚠️决策确认节点 → H5 |
| 技术债严重 | → H4债务围栏 |

## 问题路由（什么情况用什么框架）

遇到问题时，先定位类型，再选择对应框架：

| 问题类型 | 推荐框架/方法 |
|---------|--------------|
| **团队结构不合理，协调成本高** | Conway's Law + Team Topologies |
| **服务/模块边界模糊** | Bounded Context + SCS + H6分解边界检验 |
| **需求模糊不知从何下手** | Impact Mapping + User Story Mapping |
| **担心过度设计/YAGNI违反** | YAGNI + KISS + 框架4 |
| **技术债累积影响交付** | 技术债务篱笆 + 债务四象限 + H4债务围栏 |
| **考虑拆微服务** | 先用 H7 微服务先决条件检验 |
| **两个架构方案犹豫不决** | 张力分析 + H5 可逆性决策 |
| **架构有味道/感觉不对** | 反模式1-6 检查清单 |

**快速诊断入口**：不确定从哪开始时，先问"主要痛点是什么"——协调问题用Conway's Law；边界问题用Bounded Context；债务问题用技术债务篱笆。

## 共识框架（6个跨流派公认原则）

这些原则在大多数情况下都成立，是架构决策的锚点。

### 框架1: 强模块边界

**一句话** 内聚性↑ 耦合性↓ 模块边界清晰是长期可持续性的基础。

**证据**:
- DDD的Bounded Context：语言变化处即边界，不同BC用不同方式建模同一概念
- SCS (Simple Component System)：每个组件拥有自己的数据、逻辑、UI，禁止共享业务代码
- C4 Model：System → Container → Component → Code 四层清晰抽象

**应用** 遇到模块边界模糊、职责不清、耦合混乱时，用这个框架审视。

**局限** 在紧急救火场景中，过度强调边界可能拖慢交付。边界是长期投资，短期内可以容忍一定的混乱。

---

### ⚠️ 决策确认节点：团队重组

**在重组团队结构或调整服务所有权之前，先确认：**

1. 这次重组是为了**解决真实的协调问题**，还是"觉得应该这样做"？
2. 你是否清楚这次重组会对**现有系统架构**产生什么影响？
3. 重组后，团队之间的**协作接口**是否已经定义清楚？

**如果答案不清晰，建议先暂停，用Inverse Conway Maneuver验证。**

---

### 框架2: Conway's Law

**一句话** 组织结构决定系统结构——无论你喜欢与否。

**证据**:
- 1968年Melvin Conway原始论文"How Do Committees Invent?"
- Inverse Conway Maneuver：让系统设计跟随期望的组织结构，而非相反
- Martin Fowler明确将其作为微服务决策的核心考量

**应用** 重组团队前先问：这个团队结构会产生什么样的系统？想让系统长成什么样，就让团队先长成什么样。

**局限** Conway's Law描述的是趋势，不是宿命。有足够强的工程文化时，可以部分打破这个定律。

---

### 框架3: 技术债务篱笆

**一句话** 技术债务有利息。你不付它，它就拖垮你。

**证据**:
- Ward Cunningham 1992年OOPSLA首次提出技术债务比喻
- Martin Fowler的Design Stamina Hypothesis：忽视设计几周内越过"设计回报线"，之后生产力持续下降
- 技术债务四象限：只有"审慎的故意债务"在特定条件下合理

**应用** 高活动区域（频繁修改）零容忍债务，用篱笆围起来。稳定区域（不常修改）可以容忍适度债务。

**局限** 债务篱笆策略需要团队有足够的纪律去维护，否则篱笆本身会腐坏。

---

### 框架4: YAGNI + KISS

**一句话** 不为假定的未来需求构建。简单优于复杂。

**证据**:
- Kohavi等人的实证研究：即使经过仔细的前置分析，也只有约1/3的"为未来需求构建的功能"实际改善了它们被设计来改善的指标
- Martin Fowler：YAGNI不是"不做任何准备"，而是"不为不确定的需求做确定性准备"
- KISS原则（Keep It Simple, Stupid）在Unix哲学中的体现

**应用** 问自己：这个复杂度解决的是今天的问题，还是想象中的明天的问题？

**局限** 在高度可预测的领域（如金融合约、监管合规），某些"过度设计"可能是必要的。

---

### 框架5: 需求必须有价值且可测试

**一句话** 不管什么方法论，最终交付物必须对用户/业务有实际价值且可验证。

**证据**:
- INVEST原则的V (Valuable)：故事必须对客户有价值
- MoSCoW的M (Must-have)：没有则产品无法发布的才有最高优先级
- JTBD的核心：用户雇用产品完成一个"工作"，无法完成工作的功能没有价值

**应用** 评审需求时先问：谁会因为这个功能生活变得更好？如果答不上来，需求可能有问题。

**局限** 有些技术债务类的"内部需求"（如重构、架构升级）短期不直接面向用户，但长期有价值。需要单独追踪。

---

### 框架6: 演化需要纪律

**一句话** 演化设计不是"无设计"——缺乏对架构质量的持续关注是所有方法论失败的共同根源。

**证据**:
- Martin Fowler："Evolutionary design diverges and fails without someone maintaining design quality"
- 演化式设计的前提：持续重构 + 自动化测试 + 强模块边界
- 没有人在意设计质量时，演化设计必然失败

**应用** 选择"演化"而非"预先设计"时，先问自己：团队有足够的纪律持续维护设计质量吗？

**局限** 高度不确定的探索期项目，纪律难以建立，演化可能走向混乱。这时需要更多的前期引导。

## 决策启发式（8条条件化判断规则）

### H1: 规模阈值判断

```
< 10人团队，< 5个核心模块  →  单体或模块化单体（优先）
10-50人团队，多个域       →  考虑微服务或SCS（谨慎）
> 50人团队，复杂域        →  微服务 + 平台团队（需要成熟度）
```

超过阈值不一定要拆分；未达阈值也不一定不能用单体。阈值是警戒线，不是指令。

---

### H2: 独立部署测试

**启发式** 如果一个模块无法独立部署，它就不是一个模块。

**应用场景** 评估现有模块边界是否合理；决定新模块是否值得独立部署。

**检验问题** 能否在不影响其他模块的情况下独立发布这个模块？

---

### H3: 团队认知负荷警戒线

**启发式** 当协调成本开始指数上升时，审视团队边界。

**应用场景** 跨团队沟通越来越频繁、会议越来越多、等待时间越来越长时。

**触发条件** 满足任一：频繁的跨团队协调导致交付延迟；一个模块变更频繁引发其他模块问题；数据库成为共享依赖。

---

### H4: 债务围栏策略

```
高活动区域（核心业务，频繁修改）  →  零容忍债务
稳定区域（遗留系统，不常修改）    →  适度债务可接受，但需要围栏
实验区域（新探索，不确定）        →  可以快速试错，接受债务，后期清理
```

---

### ⚠️ 决策确认节点：架构方案选择

**在两个架构方案之间犹豫不决时，先做这个判断：**

**这个决策是可逆的吗？**

| 如果决策是... | 策略 |
|-------------|------|
| **可逆的**（API格式、UI布局、代码结构） | 延迟到有更多信息再做；先选一个能跑的 |
| **不可逆的**（数据模型、核心技术选型） | 需要前置分析，但不追求完美答案 |

**把确定性留给真正不确定的事情。**

**在做决定之前，确认：**
1. 你是否清楚这个决策的**可逆性**？（如果不确定，倾向于延迟）
2. 你是否有足够的**信息**做出不可逆决策？
3. 你是否预留了**回滚路径**？（如果没有，倾向于可逆方案）

---

### H5: 可逆性决策优先级

| 决策类型 | 策略 |
|---------|------|
| **可逆决策**（API格式、UI布局、代码结构） | 延迟到有更多信息再做 |
| **不可逆决策**（数据模型、核心技术选型） | 需要前置分析，但不追求完美答案 |

**核心原则** 把确定性留给真正不确定的事情。

---

### H6: 分解边界检验

**服务间大量同步调用** → 边界可能不对（分布式单体预警）

**需要同时修改多个服务才能实现单一业务功能** → 边界可能不对

**分布式事务覆盖多个服务** → 架构复杂度已超过收益

---

### ⚠️ 决策确认节点：微服务拆分

**在建议拆分微服务之前，先确认以下问题：**

1. 你的团队是否**真的需要**独立部署和独立伸缩？（不是技术偏好，而是业务刚需）
2. 你的团队是否**已经准备好**承受分布式系统的复杂度？
3. 你是否清楚**业务边界**在哪里？（不是技术边界，而是领域边界）

**如果以上任一回答为"不确定"**，请先执行 H7 先决条件检验，即使结果是正面的也建议谨慎推进。

---

### H7: 微服务先决条件检验

微服务不是目标，是能力。在决定拆分微服务前，确认以下先决条件：

1. ✅ 团队足够大（>50人）且有明确的团队边界
2. ✅ 有成熟的CI/CD和自动化部署能力
3. ✅ 有快速配置服务的能力（服务 provisioning < 1小时）
4. ✅ 有完善的监控和可观测性基础设施
5. ✅ 业务边界已经通过其他方式（DDD/事件风暴）清晰定义

**任一不满足** → 优先考虑模块化单体或SCS架构。

---

### H8: 标准化 vs 定制化平衡

| 条件 | 推荐 |
|------|------|
| 小团队，系统复杂度低 | 标准技术栈 + 约定优于配置 |
| 大团队，多个独立域 | 标准化核心 + 允许域特定例外 |
| 团队文化和成熟度高 | 更多定制化自主 |
| 高度监管/合规环境 | 更多标准化以满足合规要求 |

## 反模式（6种架构分解失败模式）

遇到这些模式时，需要警惕并考虑重构。

### 反模式1: 分布式大泥球

**特征** 服务被拆分成独立代码库，但仍然紧密耦合——需要同时部署、同步调用、共享数据库。

**典型表现** 改一个服务必须改其他服务；一个服务挂了其他服务也跟着挂；部署必须协调。

**根本原因** 用微服务的姿势干了微内核的事，获得了所有分布式复杂度而没有获得独立部署的好处。

---

### 反模式2: 过早微服务

**特征** <5人的小团队管理着数十个服务；每个服务只有几百行代码。

**典型表现** 大部分时间在管理服务间依赖和部署配置，而非写业务逻辑；每个服务的"微"已经失去意义。

**根本原因** 把微服务当作目标而非能力；在团队没有准备好承受分布式复杂度时就强行拆分。

---

### 反模式3: 能力不足的微服务

**特征** 开始微服务化但没有相应的DevOps能力。

**典型表现** 服务部署靠手动；配置靠修改服务器文件；出问题靠重启；没有监控不知道慢在哪。

**根本原因** 微服务需要基础设施先行。把服务拆出来但基础设施没跟上，每个服务都变成一个运维噩梦。

---

### 反模式4: 泥球服务

**特征** 服务内部代码纠缠，没有清晰的模块边界——服务层面拆了，但内部还是一个大泥球。

**典型表现** 改一个功能要改很多文件；很难找到某个逻辑在哪里；测试覆盖率低且脆弱。

**根本原因** 把"服务边界"当成"模块边界"的替代品，但实际上服务内部也需要模块化。

---

### 反模式5: 过度抽象

**特征** 为小问题创建复杂抽象层；引入了本来不存在的复杂性。

**典型表现** 只有两层时就能解决的问题，非要引入中间层、插件化、策略模式；抽象层比实现层还复杂。

**根本原因** YAGNI违反。"万一日后需要"是架构腐化的开始。

---

### 反模式6: Conway定律违背

**特征** 架构设计与组织结构严重不匹配——团队边界和服务边界冲突。

**典型表现** 每个功能都需要跨多个团队协调；团队说"这不是我们负责的"；依赖管理变成政治问题。

**根本原因** 架构设计时没有考虑团队现实，或者组织结构已经变化但架构没有跟进。

---

## 表达DNA（框架使用时的思维风格）

这个框架在给出建议时遵循以下风格：

| 维度 | 风格 | 说明 |
|------|------|------|
| **确定性** | **条件性断言** | 除非是跨流派共识，否则说"在X条件下，Y倾向于成立"而非"必须Y" |
| **结构** | **先结论后分析** | 先给出判断（好/不好/条件成立），再解释原因 |
| **平衡** | **呈现张力** | 不掩盖权衡，给出不同选择的边界条件 |
| **证据** | **指向来源** | 判断基于一手工程实践（Martin Fowler、Conway、Team Topologies等）而非通用原则 |
| **触发器** | **用数字锚定** | 10人/50人等阈值来自工程实践，不是理论推导 |

**不这样做**:
- 不说"微服务永远是好的"或"单体永远是好的"
- 不把条件性判断说成绝对真理
- 不忽略争议和未解决的张力

## 流派对比（各框架的适用条件）

### 架构分解流派

| 流派 | 核心主张 | 最佳适用 | 不适用 |
|------|---------|---------|-------|
| **DDD/Bounded Context** | 业务边界优先于技术边界；语言统一性是核心 | 复杂业务域，多团队，战略技术决策 | 简单CRUD系统，小团队 |
| **SCS (Simple Component System)** | 数据+逻辑+UI三元一体；禁止共享数据库 | 中等规模，需要清晰所有权 | 需要高数据冗余的场景 |
| **传统模块化** | 静态代码结构优先于运行时边界 | 单一团队，小型系统 | 需要独立部署的分布式系统 |
| **微服务** | 独立部署和伸缩是核心价值 | 大规模，多团队，高伸缩性需求 | 小团队，不确定边界，DevOps不成熟 |

## 核心张力（4对不可调和的分歧）

这些分歧背后没有"正确答案"——认识到它们是真实的张力，比强行调和更有价值。

### 张力1: Upfront Design vs Evolutionary Design

**左侧（Upfront）**: 预分析可以识别明显的高层风险和依赖；忽视早期设计会导致后期代价高昂的返工。

**右侧（Evolutionary）**: 预测未来需求本质不可靠；Kohavi等实证研究支持YAGNI；"设计耐力假说"表明好设计维持恒定生产力。

**边界条件**: 高确定性领域（金融合约、监管合规）→ Upfront；高不确定性探索期 → Evolutionary。

---

### 张力2: Full Ownership vs Shared Platform

**左侧（Full Ownership）**: 端到端责任感驱动长期好决策；减少跨团队依赖；开发者直接接触生产环境。

**右侧（Shared Platform）**: 专业化平台加速交付；避免重复造轮子；复杂子系统需要专家处理。

**边界条件**: 小团队（<20人）→ Full Ownership；大团队（>50人）+ 明确平台需求 → Shared Platform。

---

### 张力3: 速度 vs 质量

**左侧（速度）**: Move Fast是竞争优势；完美是良好的敌人；快速验证假设比设计完美更重要。

**右侧（质量）**: Design Stamina Hypothesis；忽视质量几周内越过回报线；技术债务利息会压垮团队。

**边界条件**: 初创期/生死攸关的速度 → 速度优先；金融/关键基础设施/高活动区域 → 质量优先。

---

### 张力4: 标准化 vs 上下文定制

**左侧（标准化）**: 约定优于配置加速启动；统一工具链降低认知负荷；新成员更快上手。

**右侧（上下文定制）**: 不同域有不同需求；固执框架可能引入不必要的复杂性；异构技术栈是合理的。

**边界条件**: 小团队/初期 → 标准化；大团队/成熟组织 → 标准化核心 + 例外机制。

## 失败预防（什么情况不该用这个框架）

以下信号表明当前问题**不适合**用此Skill：

| 信号 | 说明 | 正确处理 |
|------|------|---------|
| **问题已到代码实现层** | 具体类/函数设计、代码逻辑问题 | 应使用编程类Skill |
| **问题是纯语法/语言选择** | 如"这个Python语法怎么写"、"Rust好还是Go好" | 不触发架构思维框架 |
| **询问主观偏好无客观标准** | 如"你更喜欢哪种风格"、"哪个看起来更优雅" | 框架无法给出客观答案 |
| **场景极度未知且无先例** | 框架基于历史模式，全新范式仅有参考价值 | 标注不确定性，给出框架的边界条件 |
| **组织政治问题伪装成技术问题** | 如"团队不想用微服务是因为政治原因" | 框架只能处理技术维度，需标注 |

### Fallback路径

当框架不适用或遇到异常时：

| 情况 | Fallback策略 |
|------|-------------|
| **无法判断规模**（不知道多少人） | 先问团队规模；如果用户也不知道，假设单体优先 |
| **信息不足无法做决策** | 延迟决策，优先收集信息；用H5可逆性原则：能晚做就晚做 |
| **两个框架给出矛盾建议** | 用张力分析处理；呈现矛盾而非和稀泥 |
| **用户拒绝确认节点** | 直接给出建议但明确标注风险；不强制确认 |
| **框架判断与直觉冲突** | 标注"框架建议是X，但你的直觉是Y，最终取决于你" |

## 诚实边界（框架的局限性）

这个框架基于工程实践和一手研究，存在以下固有局限：

1. **无法预测全新问题的反应** — 框架基于历史模式，对真正前所未有的技术范式转变（AI驱动的开发、后云原生等）无法给出确定答案

2. **无法替代架构判断力** — 框架是工具不是替代品，最终判断需要经验和对特定上下文的理解

3. **规模阈值是经验法则** — 10人/50人等数字是行业经验，真实边界因团队成熟度、行业、地理分布、文化而差异巨大

4. **DevOps成熟度难以量化** — "成熟的CI/CD能力"是模糊概念，实际成熟度需要具体评估

5. **信息完整度影响质量** — 当只接触到项目局部信息（二手描述、非技术因素影响）时，判断质量会下降

# Code Agent 评审框架

> 「先体感，后落点；能说清楚理由，才算真的评过；留下证据，才算真的交过底。」

## 核心原则

**先体感，后落点**：不强制逐行审计；以「作为用户的实际体验」为主。
**先选档位，再微调**：避免过度量化导致评审负担。
**每个给分点必须写理由**：无理由不评分。
**扣分必须留证据**：每个扣分点对应一条证据记录，存入模型专属目录。

---

## 快速打分流程

### Step 1: 选档位
**输入**：用户的评审请求（描述 agent 的工作内容）
**输出**：初选档位（9-10 / 7-8 / 5-6 / 3-4 / 0-2）
**依据**：参考档位参考表，凭体感选择

### Step 2: 确认检查点
**输入**：Step 1 的档位 + 用户补充信息
**输出**：已确认的检查项（是否验证/核心验收点/特殊要求/被评审模型名称）
**动作**：
- 向用户确认，未确认则用默认值
- **提取模型名称**（关键）：从用户描述中识别被评审模型。若未提及，询问「请告知被评审的模型名称（如 claude-opus-4、gpt-4o、deepseek-v3），用于证据归档」。若用户拒绝提供，使用 `unknown-model`

### Step 3: 查封顶
**输入**：Step 2 结果
**输出**：封顶状态（是/否 + 原因）
**规则**：
- 核心需求未完成 → 总分 ≤ 6
- 不可运行/无法验证 → 总分 ≤ 4
- 明显跑题 → 总分 ≤ 2

### Step 4: 三维度落点（含结构化扣分）
**输入**：用户提供的 agent 工作描述 + 代码（如有）
**输出**：各维度得分 + 结构化扣分记录
**规则**：每项扣分必须填写以下结构：

```
扣分项：
  维度: 需求完成度 | 代码质量 | 轨迹质量
  扣分: -N分
  原因: [具体描述，一句话说清楚为什么扣]
  位置: [文件:行号 | 功能模块 | 轨迹步骤编号 | N/A]
  严重度: high | medium | low
  证据类型: code_snippet | test_output | error_log | screenshot | description
```

### Step 5: 写理由
**输入**：Step 3 + Step 4 结果
**输出**：五行评分总结（必填，必须保留换行）
**模板**：

```markdown
总分 X/10
需求 A/5：[一句话说明需求完成情况与主要扣分点]；
代码 B/3：[一句话说明代码质量与主要风险]；
轨迹 C/2：[一句话说明执行过程可信度]。
封顶：是/否，[封顶原因；未封顶则写“未触发封顶”]。
```

**规则**：
- 评分总结必须出现在评审结果中，且优先于长篇扣分说明。
- 每行只写该维度最关键的结论，避免把完整证据清单塞进总结。
- 命中封顶时，`封顶` 行必须写清触发条件（如 `不可运行`、`核心需求未完成`、`跑题`）。

### Step 6: 输出评审卡片
**输入**：Step 5 结果
**输出**：完整评审卡片（见下方格式）

### Step 7: 证据留档
**输入**：Step 4 的所有扣分项 + 用户提供的代码/截图/日志
**输出**：证据目录 + `DEDUCTIONS.md`
**执行顺序**：

```
1. 构造目录路径：evaluations/{model-name}/{YYYYMMDD-HHMMSS}/
   - model-name：来自 Step 2 提取的模型名，小写+连字符（如 claude-opus-4、gpt-4o）
   - 时间戳：评审执行时间
   - 如目录已存在，追加后缀 -2 / -3

2. 创建目录结构：
   evaluations/
     {model-name}/
       {timestamp}/
         DEDUCTIONS.md    ← 主报告
         evidence/        ← 原始证据文件

3. 生成 DEDUCTIONS.md（见下方模板）

4. 保存用户提供的证据：
   - 代码片段 → evidence/snippet-{N}.{ext}
   - 错误日志 → evidence/error-{N}.log
   - 截图描述 → evidence/screenshot-{N}.md（文字描述截图内容）
   - 测试输出 → evidence/test-output-{N}.txt

5. 告知用户：「证据已归档至 evaluations/{model-name}/{timestamp}/」
```

**异常处理**：
- 无法创建目录（权限/路径问题）→ 在评审卡片末尾附加「证据归档失败：{原因}」，仍输出 DEDUCTIONS.md 内容到控制台
- 用户未提供任何代码/日志 → evidence/ 目录留空，DEDUCTIONS.md 证据字段填「用户未提供，依据描述推断」

## 三维度评分标准

### 维度1: 需求完成度（0-5）「能不能用」

| 分数 | 标准 |
|------|------|
| 5 | 核心需求全部满足；主流程顺畅；边界/异常场景基本覆盖；UI/可视化达到可上线标准 |
| 4 | 核心需求满足；有少量非关键缺口或小 Bug，但不影响主流程 |
| 3 | 主流程勉强跑通；缺关键细节/验收点；需手动补救才能用 |
| 2 | 只完成一部分；核心验收点没过或经常出错 |
| 1 | 接近跑偏；和需求匹配度很低 |
| 0 | 未交付有效结果/不可运行/完全不相关 |

### 维度2: 代码质量（0-3）「敢不敢合、好不好维护」

不要求深审，抓「明显雷点」和「可维护性」。

| 分数 | 标准 |
|------|------|
| 3 | 结构清晰；改动克制；命名/边界处理合理；无明显安全/性能坑；有必要的测试或自检 |
| 2 | 整体可维护；有些瑕疵（重复、轻微坏味道、类型/异常处理一般），但风险可控 |
| 1 | 能用但明显脆弱（硬编码、耦合重、缺关键校验、潜在回归点多） |
| 0 | 明显不安全/高风险/不可维护（注入风险、到处吞异常、关键逻辑错误） |

**常见扣分点（看到就扣，并记录至 DEDUCTIONS.md）：**
- 安全风险（命令/SQL 注入、任意文件读写、泄露密钥、绕过鉴权）→ 直接 0-1，严重度 🔴 high
- 大范围无关改动（不必要格式化整库、过度重构导致 review 困难）→ 扣 0-2，严重度 🟡 medium
- 缺关键保护（吞异常不处理、无超时/重试、无输入校验、竞态明显）→ 扣 0-2，严重度 🟡 medium

### 维度3: 轨迹质量（0-2）「过程是否可信、可复用、可协作」

| 分数 | 标准 |
|------|------|
| 2 | 步骤可复现；会先澄清关键不确定点；解释取舍；工具/命令使用克制且安全；有验证闭环 |
| 1 | 过程基本顺；但有跳步、解释不足、偶尔走弯路或验证不充分 |
| 0 | 过程混乱或不可信（编造结果、未验证却宣称完成、反复无效尝试、忽略关键风险/约束） |

**提示**：轨迹质量是可信可理解，不等同于流程长度。

## 总分结构（10分）

```
总分 = 需求完成度（0-5）+ 代码质量（0-3）+ 轨迹质量（0-2）
```

### 总分封顶规则（全局规则，必须执行）

以下任一情况命中，总分**必须封顶**，不允许被其他维度抬上去：

| 命中条件 | 封顶上限 |
|---------|---------|
| 核心需求未完成 | 总分 ≤ 6 |
| 结果不可运行/无法验证，且未提供清晰复现与修复指引 | 总分 ≤ 4 |
| 明显跑题 | 总分 ≤ 2 |

## 档位与维度分不一致的处理

档位是「体验预判」，维度分是「落地算分」。两者不一致很常见。

**处理规则：**
1. 先算维度分得总分
2. 用档位做一次 sanity check
3. 若总分落在档位区间内 → 直接采用
4. 若偏离档位 2 分及以上 → 二选一：
   - **调整档位**：承认体感预判有偏差，以维度事实为准
   - **复核维度分**：若仍坚持原档位，必须写清楚「为何维度分没体现该体验」

|--------|--------|-----------|
| 是否已验证结果可运行？ | 未验证 | 可改为「已验证」并提供验证方式 |

| 是否有特殊质量要求？ | 无 | 可补充（如安全、性能、兼容性） |
| 被评审的模型名称？ | unknown-model | 必填，用于证据目录命名 |

## 产物效果证明要求

每个评审必须附带可视化效果证明，用于支撑评分结论：

| 验证方式 | 适用场景 | 说明 |
|---------|---------|------|
| 运行截图 | 功能可运行 | 展示核心功能执行结果 |
| 前后对比图 | Bug修复 | 对比修复前后的行为差异 |
| 测试报告 | 有测试的代码 | 展示测试覆盖率/通过率 |
| 错误日志 | 不可运行的情况 | 提供错误信息便于定位 |

**验证状态判定：**
- **已验证**：用户提供了上述证明材料 → 正常评分
- **未验证**：无法提供证明 → 需求完成度上限4分，轨迹质量上限1分
- **部分验证**：只提供了部分验收点的证明 → 酌情扣分

**处理逻辑**：
1. 评审开始时询问用户验证状态
2. 若用户未提供证明，在输出中标注「未验证，默认扣分」
3. 若用户后来补提供了证明，可追溯修正评分

## 证据目录结构说明

```
evaluations/                    ← 项目根目录下自动创建
  {model-name}/                 ← 如 claude-opus-4、gpt-4o、deepseek-v3
    {YYYYMMDD-HHMMSS}/          ← 评审时间戳
      DEDUCTIONS.md             ← 主报告（扣分清单+Bug清单）
      evidence/                 ← 原始证据
        snippet-1.py            ← 用户提供的代码片段
        error-1.log             ← 错误日志
        test-output-1.txt       ← 测试输出
        screenshot-1.md         ← 截图内容的文字描述
  {model-name-2}/               ← 其他模型的评审记录独立存放
    ...
```

**模型名称规范**：
- 统一小写 + 连字符（如 `claude-opus-4`，非 `Claude Opus 4`）
- 不含特殊字符（空格、斜杠等用 `-` 替代）
- 若同一模型多次评审，时间戳自然区分，无需改目录名

## 常见问题

**Q1: 需求都做完了但代码很烂，能给 9 吗？**
A: 通常不行。需求满分只占 5 分，代码质量会拉低；若存在明显安全/维护风险，代码质量应打 0-1，并在 DEDUCTIONS.md 记录为 🔴 high 扣分项。

**Q2: 轨迹写得很漂亮但结果没做出来，能给高分吗？**
A: 不能。封顶规则优先于加分项；核心需求没完成，最多 6 分。

**Q3: 用户不知道模型名称怎么办？**
A: 使用 `unknown-model` 作为目录名，在 DEDUCTIONS.md 顶部注明「模型名称未知」。

**Q4: 证据目录应该创建在哪里？**
A: 默认在当前工作目录下创建 `evaluations/`。若用户明确指定其他路径，以用户指定为准。

## 🛑 反例与黑名单(Anti-Patterns)

**评审者必看 — 这些是已知会让评审失效的行为,任一命中应立即修正或回滚。**

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|------------|---------|
| A1 | **跳过档位直接给维度分** | 失去 sanity check,维度分容易虚高 | Step 1 先选档位 → Step 4 再算维度分 → 两者比对 |
| A2 | **"需求满分+代码很烂" 给了 9-10** | 需求 5 + 代码 0 + 轨迹 2 = 7 上限;9-10 必然错算 | 严格按 5+3+2=10 结构加,绝不取档位最大值 |
| A3 | **"轨迹漂亮"就给高分** | 封顶规则强制:核心需求未完成 ≤ 6,跑题 ≤ 2,不可运行 ≤ 4 | 先算总分 → 再看封顶 → 封顶优先于其他维度 |
| A4 | **扣分不写证据** | "扣 1 分因为代码不好" → 无代码引用,等于没扣 | 每条扣分必须有"位置(文件:行号)+ 原因 + 严重度"三要素 |
| A5 | **用 unknown-model 但不留说明** | 后续用户找不到评审对象,失去可追溯性 | 强制在 DEDUCTIONS.md 顶部注明"模型名称未知" |
| A6 | **证据目录建在临时目录** | 进程结束后被清理,失去证据 | 默认工作目录 `evaluations/{model}/{ts}/`,不要用 `/tmp` |
| A7 | **手写 DEDUCTIONS.md 内容到控制台后忘记归档** | 评审看起来完成了,实际没留底 | Step 7 异常处理:归档失败时,先输出 DEDUCTIONS.md 全文到评审卡片末尾,再让用户选择 |
| A8 | **"未验证" 不触发上限** | 需求 4 + 轨迹 1 → 用户不知道有上限 | 评审开头主动询问验证状态,无证明时强制标注"未验证" |
| A9 | **评分总结塞完整证据清单** | 五行总结失去"高密度结论"的价值,读者抓不到主线 | 每行只写该维度最关键的 1 句结论,详细证据放 DEDUCTIONS.md |
| A10 | **封顶时不写触发条件** | "是,触发封顶" → 用户不知道为什么 | 必须写"是,核心需求未完成" / "是,不可运行" / "是,跑题" |

# Code Agent 评审框架

> 「先体感，后落点；能说清楚理由，才算真的评过。」

## 核心原则

**先体感，后落点**：不强制逐行审计；以「作为用户的实际体验」为主。
**先选档位，再微调**：避免过度量化导致评审负担。
**每个给分点必须写理由**：无理由不评分。

---

## 快速打分流程

### Step 1: 选档位
**输入**：用户的评审请求（描述agent的工作内容）
**输出**：初选档位（9-10 / 7-8 / 5-6 / 3-4 / 0-2）
**依据**：参考档位参考表，凭体感选择

### Step 2: 确认检查点
**输入**：Step 1的档位 + 用户补充信息
**输出**：已确认的检查项（是否验证/核心验收点/特殊要求）
**动作**：向用户确认，未确认则用默认值

### Step 3: 查封顶
**输入**：Step 2结果
**输出**：封顶状态（是/否 + 原因）
**规则**：
- 核心需求未完成 → 总分 ≤ 6
- 不可运行/无法验证 → 总分 ≤ 4
- 明显跑题 → 总分 ≤ 2

### Step 4: 三维度落点
**输入**：用户提供的agent工作描述 + 代码（如有）
**输出**：各维度得分（需求0-5、代码0-3、轨迹0-2）
**规则**：每项必须写扣分理由

### Step 5: 写理由
**输入**：Step 3 + Step 4结果
**输出**：一句话评审理由（必填）
**模板**：`总分 X/10：需求 A/5（...）；代码 B/3（...）；轨迹 C/2（...）。封顶：是/否（原因...）。`

### Step 6: 输出评审卡片
**输入**：Step 5结果
**输出**：完整评审卡片（见下方格式）

### 档位参考

| 档位 | 体感 | 你是否愿意直接合入 |
|------|------|------------------|
| 9-10 | 惊喜，几乎可直接上线 | 非常愿意复用该 agent 工作方式 |
| 7-8 | 满意，少量瑕疵 | 愿意自己收尾（半小时内） |
| 5-6 | 勉强可用，需较多补救 | 不太敢直接合 |
| 3-4 | 明显不行，主问题没解决 | 只能当作局部参考 |
| 0-2 | 基本失败/跑题/不可信 | 无有效交付 |

## 三维度评分标准

### 维度1: 需求完成度（0-5）「能不能用」

| 分数 | 标准 |
|------|------|
| 5 | 核心需求全部满足；主流程顺畅；边界/异常场景基本覆盖；UI/可视化达到可上线标准 |
| 4 | 核心需求满足；有少量非关键缺口或小 Bug，但不影响主流程 |
| 3 | 主流程勉强跑通；缺关键细节/验收点；需手动补救才能用 |
| 2 | 只完成一部分；核心验收点没过或经常出错 |
| 1 | 接近跑偏；和需求匹配度很低 |
| 0 | 未交付有效结果/不可运行/完全不相关 |

### 维度2: 代码质量（0-3）「敢不敢合、好不好维护」

不要求深审，抓「明显雷点」和「可维护性」。

| 分数 | 标准 |
|------|------|
| 3 | 结构清晰；改动克制；命名/边界处理合理；无明显安全/性能坑；有必要的测试或自检 |
| 2 | 整体可维护；有些瑕疵（重复、轻微坏味道、类型/异常处理一般），但风险可控 |
| 1 | 能用但明显脆弱（硬编码、耦合重、缺关键校验、潜在回归点多） |
| 0 | 明显不安全/高风险/不可维护（注入风险、到处吞异常、关键逻辑错误） |

**常见扣分点（看到就扣）：**
- 安全风险（命令/SQL 注入、任意文件读写、泄露密钥、绕过鉴权）→ 直接 0-1
- 大范围无关改动（不必要格式化整库、过度重构导致 review 困难）→ 扣 0-2
- 缺关键保护（吞异常不处理、无超时/重试、无输入校验、竞态明显）→ 扣 0-2

### 维度3: 轨迹质量（0-2）「过程是否可信、可复用、可协作」

| 分数 | 标准 |
|------|------|
| 2 | 步骤可复现；会先澄清关键不确定点；解释取舍；工具/命令使用克制且安全；有验证闭环 |
| 1 | 过程基本顺；但有跳步、解释不足、偶尔走弯路或验证不充分 |
| 0 | 过程混乱或不可信（编造结果、未验证却宣称完成、反复无效尝试、忽略关键风险/约束） |

**提示**：轨迹质量是可信可理解，不等同于流程长度。

## 总分结构（10分）

```
总分 = 需求完成度（0-5）+ 代码质量（0-3）+ 轨迹质量（0-2）
```

### 总分封顶规则（全局规则，必须执行）

以下任一情况命中，总分**必须封顶**，不允许被其他维度抬上去：

| 命中条件 | 封顶上限 |
|---------|---------|
| 核心需求未完成 | 总分 ≤ 6 |
| 结果不可运行/无法验证，且未提供清晰复现与修复指引 | 总分 ≤ 4 |
| 明显跑题 | 总分 ≤ 2 |

---

## 评估理由模板

```
总分 X/10：需求 A/5（...）；代码 B/3（...）；轨迹 C/2（...）。封顶：是/否（原因...）。
```

### 示例

- 总分 8/10：需求 4/5（核心验收过，边界少一个）；代码 3/3（改动克制，无明显风险）；轨迹 1/2（验证记录偏少）。**封顶：否。**
- 总分 6/10：需求 3/5（主流程通但缺关键验收点）；代码 2/3（有重复与硬编码）；轨迹 1/2（过程可复现）。**封顶：是（核心验收未完成）。**
- 总分 4/10：需求 2/5（结果不可运行且缺复现指引）；代码 1/3（潜在回归点多）；轨迹 1/2。**封顶：是（不可验证）。**

## 档位与维度分不一致的处理

档位是「体验预判」，维度分是「落地算分」。两者不一致很常见。

**处理规则：**
1. 先算维度分得总分
2. 用档位做一次 sanity check
3. 若总分落在档位区间内 → 直接采用
4. 若偏离档位 2 分及以上 → 二选一：
   - **调整档位**：承认体感预判有偏差，以维度事实为准
   - **复核维度分**：若仍坚持原档位，必须写清楚「为何维度分没体现该体验」

## 评审前检查点（用户确认）

评审开始前，**向用户确认以下信息**（如未提供则默认）：

| 检查项 | 默认值 | 用户可修改 |
|--------|--------|-----------|
| 是否已验证结果可运行？ | 未验证 | 可改为「已验证」并提供验证方式 |
| 核心验收点是什么？ | 用户描述 | 可补充具体验收点 |
| 是否有特殊质量要求？ | 无 | 可补充（如安全、性能、兼容性） |

## 产物效果证明要求

每个评审必须附带可视化效果证明，用于支撑评分结论：

| 验证方式 | 适用场景 | 说明 |
|---------|---------|------|
| 运行截图 | 功能可运行 | 展示核心功能执行结果 |
| 前后对比图 | Bug修复 | 对比修复前后的行为差异 |
| 测试报告 | 有测试的代码 | 展示测试覆盖率/通过率 |
| 错误日志 | 不可运行的情况 | 提供错误信息便于定位 |

**验证状态判定：**
- **已验证**：用户提供了上述证明材料 → 正常评分
- **未验证**：无法提供证明 → 需求完成度上限4分，轨迹质量上限1分
- **部分验证**：只提供了部分验收点的证明 → 酌情扣分

**处理逻辑**：
1. 评审开始时询问用户验证状态
2. 若用户未提供证明，在输出中标注「未验证，默认扣分」
3. 若用户后来补提供了证明，可追溯修正评分

## 常见问题

**Q1: 需求都做完了但代码很烂，能给 9 吗？**
A: 通常不行。需求满分只占 5 分，代码质量会拉低；若存在明显安全/维护风险，代码质量应打 0-1。

**Q2: 轨迹写得很漂亮但结果没做出来，能给高分吗？**
A: 不能。封顶规则优先于加分项；核心需求没完成，最多 6 分。
